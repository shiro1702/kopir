# 13 — Hotfix: гонка при загрузке 5 файлов скопом

| | |
|---|---|
| **Спринт** | 5 |
| **Статус** | ✅ hotfix |
| **Приоритет** | P0 |
| **Feature** | FILE-01, UX-15 |
| **Зависимости** | Sprint 0.2 batch flow, Sprint 1 status messages |
| **Оценка** | 0.5–1 день |

## Проблема

Если пользователь пересылает в бот 3–5 документов почти одновременно, входящие webhook/update-события обрабатываются параллельно. В результате текущий batch-flow может читать старое состояние пачки раньше, чем соседний файл успел записаться в БД.

Симптомы:

- у нескольких файлов одинаковый `batchIndex`
- карточки показывают неверный текст `Файлов: 1 из 5`
- лимит `BATCH_MAX_FILES=5` иногда пробивается
- `totalAmountKopeks` и reply/inline-клавиатуры пересчитываются в неправильном порядке
- при быстрой серии документов UX выглядит «скачущим», хотя сами файлы могут сохраниться

## Корень проблемы

Сейчас обработчик документа делает последовательность `count -> nextIndex -> create order -> recalculate`, но сам обработчик вызывается конкурентно для каждого файла.

Уязвимые места:

1. Проверка лимита и расчёт индекса происходят вне общей блокировки на пользователя/пачку.
2. `getNextBatchIndex()` считает `max(batchIndex) + 1`, не защищая слот от параллельной записи.
3. Обновление totals и клавиатур идёт после каждого файла отдельно, поэтому пачка «дёргается» во время серии апдейтов.

## Цель хотфикса

Сделать загрузку нескольких файлов подряд идемпотентной и предсказуемой:

- 5 файлов подряд всегда становятся индексами `1..5` без дублей
- лимит пачки не нарушается
- сумма и тексты сходятся с фактическим составом пачки
- Telegram и MAX ведут себя одинаково

## Решение

### A. Сериализация обработки на пользователя / чат

- [x] **13.1** Ввести `withUserBatchLock(...)` для bot-flow:
  - ключ минимум: `platform + externalUserId`
  - scope: критическая секция от получения/создания collecting batch до `order.create`
- [x] **13.2** Не пускать второй `handleDocument()` в критическую секцию, пока первый не завершил расчёт слота

Для single-instance среды достаточно in-memory очереди. Если появится несколько serverless-инстансов, заменить реализацию lock на DB advisory lock или Redis без смены внешнего контракта.

### B. Атомарный слот `batchIndex`

- [x] **13.3** Перенести связку:
  - `countActiveBatchOrders(batch.id)`
  - проверка `maxFiles`
  - `getNextBatchIndex(batch.id)`
  - `order.create(...)`
  в одну транзакционную операцию
- [x] **13.4** Брать lock на `OrderBatch` перед выдачей следующего индекса
- [x] **13.5** Как safety net: предусмотреть retry при конфликте индекса, если позже появится DB constraint

### C. Стабилизация UX при серии файлов

- [x] **13.6** Не считать batch broken только из-за того, что 2–3 файла ещё в обработке
- [x] **13.7** Обновлять batch keyboard после успешного добавления файла, но без лишних дубликатов сообщений
- [x] **13.8** Проверить, что текст `Файлов: N из 5` берётся из фактического `batchIndex`, а не из устаревшего снимка состояния

## Не в scope

- Буферизация Telegram `media_group_id` как отдельной сущности
- Новый экран «Моя пачка»
- Переписывание batch-flow под Mini App API
- Полная переделка очередей на Redis / WebSocket

## Критерии приёмки

- [x] 5 PDF, отправленные подряд одним действием, дают индексы `1,2,3,4,5`
- [x] При 6-м файле пользователь получает штатное сообщение про лимит, без создания лишнего `Order`
- [x] Сумма пачки после серии файлов равна сумме активных заказов в БД
- [x] После race-теста нет дублей `batchIndex` внутри одного `batchId`
- [x] Telegram и MAX проходят один и тот же сценарий без расхождений в статусах

## Проверка / smoke

- [x] Telegram: переслать 5 PDF одним действием
- [x] Telegram: переслать 6 PDF и убедиться, что шестой отклонён
- [x] MAX: отправить 3+ файлов подряд
- [x] Повторить тест на batch с уже выбранной точкой и без выбранной точки

## Заметки

- Это именно **hotfix стабильности**, а не новая продуктовая фича.
- Если нужен самый быстрый безопасный вариант, сначала делаем per-user lock, затем усиливаем транзакцией и smoke-тестом.
